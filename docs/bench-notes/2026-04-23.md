## What changed this week

- **Scoring pipeline refactor** (engine/src/scoring.rs, engine/src/search.rs): `ScoredMatch` now holds `(idx, score, title: &str)` instead of cloning a full `Candidate` on every heap push. `Candidate` clones happen only for the final top-K rather than every pushed entry.
- **Rerank reuses `IndexedCandidate.title_search`** instead of re-running `normalize_for_search` per reranked candidate.
- Net effect: hot-path allocations down substantially, visible most clearly in the empty-query browse path where every candidate passed the score threshold under the old code.

## Notes

- Biggest single win is the empty-query path (~61% faster p50 and p95). This is the path that previously cloned one `Candidate` per indexed item per keystroke; it now clones only the top-K.
- Text queries show consistent small improvements (2-13% p50). Gain is smaller because score-threshold filtering already rejected most candidates before the old code cloned them.
- Two queries show mild regressions at p50/p95 (`a"safari`, `visual` p95). Both are within typical run-to-run noise for this dataset size and single-run methodology; no code path touched by the refactor explains a regression.
- Fuzzy micro-bench is essentially flat - `look-matching` was not modified this week.
- Indexing throughput dropped (`40705/s` → `32899/s`). This is not attributable to the scoring refactor (which does not touch indexing); likely single-run variance or unrelated changes landed this week. Worth re-measuring next snapshot before treating as a real regression.
